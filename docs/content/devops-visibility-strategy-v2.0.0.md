# DevOps Visibility Strategy — v2.0.0

## Overview

Este documento define cuánto peso debe tener DevOps en la personal website `v2.0.0`.

La decisión estratégica es mantener DevOps visible, pero cambiar su papel dentro de la web.

DevOps no debe ser la identidad principal de la `v2.0.0`.

Debe funcionar como una prueba técnica complementaria que refuerza el posicionamiento principal:

> Sound Engineering · AV Networking · Technical Systems

La web no debería comunicar:

```txt
Soy DevOps junior con proyectos AV.
```

Debe comunicar:

```txt
Soy un perfil técnico AV + sonido + redes + sistemas, y además sé construir, desplegar y operar infraestructura software real.
```

Ese matiz cambia completamente la percepción profesional.

---

# 1. Decisión estratégica

La jerarquía recomendada para la web es:

```txt
Identidad principal:
Sound Engineering · AV Networking · Technical Systems

Prueba complementaria:
DevOps · Backend · Docker · CI/CD · Self-hosted infrastructure

Narrativa:
Entiendo eventos, sonido, redes e infraestructura como partes de un mismo sistema técnico.
```

DevOps debe estar presente porque diferencia el perfil, pero no debe dominar la Home, About o Services.

La parte DevOps debe reforzar esta idea:

```txt
No solo opero sistemas técnicos: también sé diseñar, automatizar, desplegar y mantener infraestructura real.
```

---

# 2. Regla principal

La regla para `v2.0.0` será:

```txt
DevOps no es el producto principal.
DevOps es la prueba de que sé pensar, construir y operar sistemas.
```

Versión más formal:

```txt
La parte DevOps, backend e infraestructura se mantiene visible como prueba de profundidad técnica, pero queda subordinada al posicionamiento principal: sonido, redes AV y sistemas técnicos para eventos y entornos de producción.
```

English version:

```txt
DevOps, backend and infrastructure remain visible as proof of technical depth, but they are subordinated to the main positioning: sound, AV networking and technical systems for live events and production environments.
```

# 3. Peso recomendado por página

## 3.1. Home

### Recommended DevOps weight

```txt
15-20%
```

## Role of DevOps on Home

En la Home, DevOps debe aparecer solo como prueba de profundidad técnica.

No debe abrir la página ni ocupar el mensaje principal.

La Home debe priorizar:

1. Sonido.
2. Redes AV.
3. Sistemas técnicos.
4. Servicios.
5. Perfil híbrido.
6. Case studies.
7. Contacto.

DevOps puede aparecer en una sección tipo:

```txt
Esta web también forma parte del sistema.
```

Pero esta sección debe ir después de servicios, perfil híbrido y/o case studies.

## Recommended Home framing

```txt
No solo hablo de sistemas: construyo, despliego y opero sistemas reales.
```

## Technologies that can appear

```txt
React
Spring Boot
Docker
NGINX
GitHub Actions
GHCR
PostgreSQL
Self-hosting
```

## What to avoid on Home

```txt
Portfolio DevOps.
Junior DevOps.
Estoy aprendiendo backend.
Estoy reorientando mi carrera hacia software.
Homelab como mensaje principal.
```

# 4. About

## Recommended DevOps weight

```txt
20-25%
```

## Role of DevOps on About

En About sí tiene sentido explicar la evolución hacia sistemas, Linux, Docker, servidores, automatización y prácticas DevOps.

Pero debe hacerse desde una idea clave:

```txt
No lo veo como un cambio de mundo, sino como una ampliación del mismo criterio técnico.
```

La página About debe explicar que la experiencia AV llevó naturalmente hacia:

- redes IP;
- sistemas AV;
- servidores;
- Linux;
- Docker;
- automatización;
- documentación;
- despliegue;
- operación;
- fiabilidad.

## Correct framing

```txt
El trabajo en AV me llevó naturalmente hacia redes, sistemas, infraestructura y automatización.
```

## Avoid framing

```txt
Dejé AV para pasarme a DevOps.
```

## Recommended About narrative

```txt
Vengo de eventos, sonido y producción audiovisual.
Ese entorno me enseñó a pensar en señal, flujo, presión, personas y fallos reales.
Con el tiempo, ese mismo criterio técnico se conectó con redes IP, servidores, Docker, automatización e infraestructura.
Hoy no separo ambos mundos: los integro dentro de una forma de trabajar orientada a sistemas técnicos fiables.
```

# 5. Services

```txt
5-10%
```

## Role of DevOps on Services

En Services no debería aparecer “DevOps” como servicio principal.

Los servicios principales son:

```txt
Sound Engineering
Monitor Engineering
Sound System Engineering
AV Networking
Technical Planning & Consulting
```

La parte DevOps puede aparecer dentro de conceptos como:

```txt
Technical Systems
Technical Infrastructure
Automation
Internal Tools
Monitoring
Documentation
Self-hosted Services
```

Pero no como un servicio tipo:

```txt
DevOps services
```

Eso rompería el posicionamiento.

## Correct framing

```txt
Infraestructura técnica, automatización y herramientas internas aplicadas a entornos AV y producción.
```

## Avoid framing

```txt
Servicios DevOps genéricos.
Desarrollo backend para cualquier cliente.
Consultoría cloud generalista.
```

# 6. Projects / Case Studies

## Recommended DevOps weight

```txt
35-40% máximo
```

## Role of DevOps on Projects

En Projects / Case Studies sí se puede enseñar claramente DevOps, backend, CI/CD, Docker, homelab y servicios propios.

Pero el portfolio debe estar equilibrado.

La mezcla recomendada es:

```txt
40% AV / sonido / eventos
30% networking / tools
30% infraestructura / backend / DevOps
```

O, para una primera versión `v2.0.0`:

```txt
1 case study AV / Event Production
1 case study Sound Engineering
1 case study AV Networking / Network Scout
1 case study Technical Infrastructure / Personal Website
1 case study Backend Automation / Contact Service
```

## Goal

Evitar que el visitante perciba:

```txt
Portfolio DevOps / software
```

Y conseguir que perciba:

```txt
Perfil híbrido AV + sonido + redes + software
```

# 8. Qué reducir o quitar

Para `v2.0.0`, conviene reducir frases como:

```txt
Estoy reorientando mi carrera hacia DevOps.
Estoy aprendiendo backend.
Estoy construyendo mi camino hacia software.
Portfolio DevOps.
Homelab como núcleo principal.
DevOps Bootcamp como identidad principal.
```

No son ideas falsas, pero posicionan peor dentro del nuevo objetivo comercial.

Pueden generar la percepción de:

```txt
Perfil junior buscando empleo DevOps.
```

Cuando la intención de la web debe ser:

```txt
Perfil técnico híbrido que une AV, sonido, redes, sistemas e infraestructura.
```

# 14. Final decision for v2.0.0

La decisión final para la `v2.0.0` queda así:

```txt
DevOps, backend and infrastructure will remain visible as supporting proof of technical depth, but they will no longer define the main identity of the website.

The main identity will be Sound Engineering, AV Networking and Technical Systems for live events, venues and production environments.

DevOps will be presented as an applied layer: automation, documentation, deployment, monitoring and operation of real technical systems.
```

Versión española:

```txt
DevOps, backend e infraestructura seguirán visibles como prueba complementaria de profundidad técnica, pero dejarán de definir la identidad principal de la web.

La identidad principal será Sound Engineering, AV Networking y Technical Systems para eventos, espacios y entornos de producción.

DevOps se presentará como una capa aplicada: automatización, documentación, despliegue, monitorización y operación de sistemas técnicos reales.
```

```txt
DevOps no desaparece de la web.
```
