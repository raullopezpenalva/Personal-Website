# Case Studies Categories — v2.0.0

## Overview

Este documento define las categorías de case studies para la personal website `v2.0.0`.

El objetivo es evitar que el portfolio parezca únicamente un portfolio DevOps/software, como ocurría en la versión anterior, y reposicionarlo hacia el perfil completo:

> Sonido · Redes AV · Infraestructura técnica

Los case studies deben demostrar que el perfil no se limita a construir webs, backends u homelabs, sino que también tiene experiencia y criterio en sonido, eventos, producción técnica, sistemas AV, redes e infraestructura.

La idea central es:

> El portfolio no debe demostrar solo que sé desplegar software. Debe demostrar que entiendo eventos, sonido, redes e infraestructura como partes de un mismo sistema técnico.

---

# 1. Objetivo de los case studies

Los case studies deben servir como prueba profesional del nuevo posicionamiento.

Deben mostrar:

- experiencia real en eventos y producción audiovisual;
- criterio de sonido;
- comprensión de sistemas de sonido;
- capacidad de planificación técnica;
- conocimiento de redes AV e infraestructura IP;
- capacidad para construir herramientas, automatizar procesos y desplegar servicios;
- mentalidad de documentación, troubleshooting y mejora continua.

No se trata de enseñar todos los proyectos posibles, sino de seleccionar casos que refuercen la narrativa profesional.

# 3. Category — AV / Event Production

## Purpose

Esta categoría demuestra experiencia real en eventos, producción técnica, coordinación y criterio de campo.

Debe venderse como capacidad de planificación, supervisión, coordinación y comprensión global del evento.

## Qué demuestra

- Experiencia en eventos reales.
- Capacidad de planificación técnica.
- Coordinación con proveedores, artistas y producción.
- Comprensión de riders, necesidades técnicas y flujos de trabajo.
- Resolución de problemas en entorno real.
- Capacidad para reducir improvisación.
- Capacidad para entender el evento como un sistema AV completo.

## Ejemplos de case studies

```txt
Technical planning for a multi-day music festival
Venue technical coordination for live events
AV production support for cultural events
Technical supervision of sound, lighting and video providers
Multi-day event production support
```

## Cómo venderlo

```txt
Planificación, coordinación y operación técnica en entornos de producción reales.
```

## Valor para cliente

```txt
Menos improvisación, mejor coordinación, riesgos detectados antes del montaje y mayor control técnico del evento.
```

## Posibles títulos públicos

```txt
Planificación técnica para festival de varios días
Coordinación técnica de sala para eventos en directo
Supervisión técnica de proveedores AV
Producción técnica para eventos culturales
```

# 4. Category — Sound Engineering

## Purpose

Esta categoría vende la base profesional de sonido.

Aquí entran FOH, monitores, sonido directo, control técnico de show, soporte a artistas, pruebas de sonido, routing y criterio musical/técnico.

## Qué demuestra

- Experiencia real en sonido directo.
- Capacidad de operar bajo presión.
- Criterio técnico y musical.
- Comunicación con artistas y producción.
- Conocimiento de consolas, patch, routing y dinámica de evento.
- Capacidad para trabajar dentro de un flujo técnico real.
- Fiabilidad durante montaje, prueba y show.

## Ejemplos de case studies

```txt
FOH sound engineering for live concerts
Monitor engineering support for live performers
Soundcheck and show operation for cultural events
Audio routing and troubleshooting in live production
Live sound support for venue programming
```

## Cómo venderlo

```txt
Soporte técnico de sonido para directos, eventos y producción audiovisual con criterio de sistema.
```

## Valor para cliente

```txt
Sonido más controlado, prueba de sonido más ordenada, mejor comunicación técnica y mayor fiabilidad durante el show.
```

## Posibles títulos públicos

```txt
FOH Sound Engineering para conciertos en directo
Soporte de monitores para artistas en directo
Operación de sonido para eventos culturales
Ruteo y troubleshooting de audio en producción real
```

# 5. Category — Sound System Engineering

## Purpose

Esta categoría eleva el perfil por encima del técnico de mesa.

El objetivo es demostrar comprensión del sistema de sonido completo: cobertura, alineamiento, subs, fills, delays, medición, coherencia y respuesta del sistema.

Aunque esta línea esté en fase de consolidación, debe aparecer como categoría estratégica porque conecta directamente con el posicionamiento de sistemas.

## Qué demuestra

- Pensamiento sistémico aplicado al audio.
- Comprensión de cobertura, alineamiento, subs, fills y delays.
- Uso de medición y análisis.
- Capacidad de verificar y optimizar sistemas.
- Enlace natural entre sonido y mentalidad de infraestructura.
- Capacidad para trabajar con datos, escucha y criterio técnico.
- Capacidad para documentar decisiones de ajuste.

## Ejemplos de case studies

```txt
Sound system verification for live event
PA alignment and measurement workflow
Subwoofer and fill alignment for venue setup
System optimization report for event production
Measurement-based PA tuning workflow
```

## Cómo venderlo

```txt
Verificación, medición y optimización de sistemas de sonido para mejorar cobertura, coherencia y fiabilidad.
```

## Valor para cliente

```txt
Sistema más coherente, menos problemas de cobertura, decisiones técnicas más objetivas y mejor soporte para FOH.
```

## Posibles títulos públicos

```txt
Verificación de sistema de sonido para evento en directo
Workflow de medición y ajuste de PA
Alineamiento de subs, fills y delays
Optimización de sistema de sonido en sala
```

## Scope note

Esta categoría debe venderse con cuidado y con alcance claro.

No debe comunicar responsabilidad total sobre sistemas diseñados o suministrados por terceros si no se ha pactado expresamente.

Puede dividirse en:

- verificación;
- medición;
- optimización;
- supervisión;
- informe técnico;
- apoyo al técnico FOH o empresa de sonido.

# 6. Category — AV Networking

## Purpose

Esta es una de las categorías más diferenciales del portfolio.

Conecta AV, redes, sistemas e infraestructura. Debe estar muy visible porque representa una ventaja competitiva clara.

La idea central es:

> Entender el evento como infraestructura.

## Qué demuestra

- Conocimiento de redes IP.
- Aplicación práctica en entornos audiovisuales.
- Dante, NDI, streaming, control, WiFi técnico y VLANs.
- Troubleshooting de conectividad.
- Documentación de red.
- Mentalidad de infraestructura aplicada a eventos.
- Capacidad para unir audio, vídeo, control y sistemas.
- Capacidad para diagnosticar problemas de campo.

## Ejemplos de case studies

```txt
Dante network design for live event
Technical WiFi and control network for production
NDI / streaming network troubleshooting
Portable network diagnostics for unmanaged environments
Network Scout field diagnostics tool
Event network topology and documentation
```

## Cómo venderlo

```txt
Diseño, despliegue y diagnóstico de redes IP para audio, vídeo, control, streaming y producción técnica.
```

## Valor para cliente

```txt
Menos conflictos IP, redes más estables, mejor separación de sistemas, troubleshooting más rápido y documentación reutilizable.
```

## Posibles títulos públicos

```txt
Diseño de red Dante para evento en directo
Red técnica WiFi para control de producción
Diagnóstico de red NDI / streaming
Network Scout: diagnóstico portátil de redes
Documentación de topología de red para evento
```

## Scope note

Esta categoría debe definir siempre qué incluye y qué no incluye.

Puede incluir:

- diseño de red;
- configuración de switches;
- despliegue en evento;
- soporte durante evento;
- troubleshooting;
- documentación;
- informe posterior.

Puede excluir:

- soporte ilimitado;
- responsabilidad sobre sistemas de terceros no documentados;
- reparación de cableado físico no pactada;
- soporte IT general de oficina;
- disponibilidad 24/7.

# 7. Category — Technical Infrastructure

## Purpose

Aquí entran la web, backends, Docker, GitHub Actions, reverse proxy, homelab, servicios self-hosted y automatización.

Pero esta categoría debe venderse como **infraestructura técnica aplicada**, no como “portfolio junior DevOps”.

Debe apoyar la narrativa AV + sistemas, no sustituirla.

## Qué demuestra

- Capacidad para construir sistemas reales.
- Automatización.
- Despliegue.
- Backend.
- Docker.
- Reverse proxy.
- CI/CD.
- Operación de servicios propios.
- Capacidad de documentar y mantener infraestructura.
- Capacidad para crear herramientas internas o servicios de soporte.

## Ejemplos de case studies

```txt
Self-hosted personal website infrastructure
Contact backend with Telegram notifications
Newsletter service
Dockerized backend services behind NGINX reverse proxy
CI/CD deployment workflow with GitHub Actions and GHCR
Network Scout backend report collector
Homelab / self-hosted services
```

## Cómo venderlo

```txt
Infraestructura, automatización y servicios técnicos aplicados a portfolio, operación y herramientas internas.
```

## Valor para cliente

```txt
Capacidad para diseñar, desplegar y mantener herramientas técnicas propias, automatizar flujos y operar servicios reales.
```

## Posibles títulos públicos

```txt
Infraestructura self-hosted de la web personal
Backend de contacto con notificaciones Telegram
Servicio de newsletter modular
Despliegue Docker + NGINX reverse proxy
Workflow CI/CD con GitHub Actions y GHCR
Colector backend para reportes de Network Scout
```

## Strategic note

Esta categoría debe ser una prueba de profundidad técnica.

No debe hacer que la web parezca exclusivamente orientada a DevOps.

La jerarquía correcta es:

```txt
Identidad principal:
Sonido + Redes AV + Sistemas técnicos

Prueba complementaria:
Infraestructura + backend + automatización + DevOps
```

# 10. Orden de prioridad para v2.0.0

Para la primera versión, no conviene publicar demasiados case studies.

La prioridad debe ser equilibrar el portfolio para que no parezca solo software.

## Priority 1 — ya maduros o fáciles de documentar

```txt
Personal Website Infrastructure
Contact Backend with Notifications
```

Estos proyectos ya existen, están maduros y son fáciles de convertir en case studies.

## Priority 2 — necesarios para equilibrar hacia AV

```txt
Technical Planning for Events
Sound Engineering for Live Events
```

Estos son importantes para que la web no siga oliendo demasiado a portfolio DevOps.

Aunque estén anonimizados, deben aparecer para reforzar la experiencia real AV.

## Priority 3 — diferencial potente

```txt
Network Scout / Portable Network Diagnostics
```

Este caso conecta perfectamente con redes, campo, troubleshooting, Linux y automatización.

```txt

```

# 12. Estructura recomendada para cada case study

Cada case study debería seguir una estructura estable:

## English structure

```txt

Role
Technical Approach
Tools / Systems
Outcome
What it demonstrates
```

## Spanish structure

```txt
Contexto
Reto
Rol asumido
Enfoque técnico
Herramientas / sistemas
Resultado
Qué demuestra
```

## Why this matters

El último punto es especialmente importante.

Cada case study debe cerrar explicando qué capacidad profesional prueba.

Ejemplo en inglés:

```txt
What it demonstrates:
This case study shows the ability to plan technical requirements, coordinate providers and understand an event as a complete AV system.
```

Ejemplo en español:

```txt
Qué demuestra:
Este case study muestra la capacidad de planificar necesidades técnicas, coordinar proveedores y entender un evento como un sistema AV completo.
```

# 13. Cómo redactar los case studies

Los case studies deben ser claros, sobrios y técnicos.

Deben evitar sonar a:

- post de ego;
- portfolio de estudiante;
- explicación interna de repo;
- CV narrativo;
- lista de herramientas sin contexto.

Deben sonar a:

- contexto real;
- problema concreto;
- rol asumido;
- criterio técnico;
- solución aplicada;
- resultado;
- capacidad demostrada.

## Fórmula recomendada

```txt
Contexto:
Dónde ocurre y qué tipo de entorno era.

Reto:
Qué problema, necesidad o incertidumbre existía.

Rol:
Qué responsabilidad asumí y qué no asumí.

Enfoque técnico:
Cómo lo abordé, qué decisiones tomé y por qué.

Herramientas / sistemas:
Qué tecnología, equipo o metodología intervino.

Resultado:
Qué se consiguió o qué se dejó preparado.

Qué demuestra:
Qué capacidad profesional prueba este caso.
```

# 14. Decisión estratégica

La decisión estratégica para `v2.0.0` queda definida así:

```txt
The portfolio will be organized around five main categories:

1. AV / Event Production
2. Sound Engineering
3. Sound System Engineering
4. AV Networking
5. Technical Infrastructure

The goal is to avoid presenting the portfolio only as a DevOps/software portfolio.

The case studies must show the full hybrid positioning:

- real AV field experience;
- sound engineering;
- event production;
- networking and AV over IP;
- infrastructure, automation and backend systems.

For v2.0.0, priority should be given to publishing a balanced set of case studies that includes at least one AV/event case and one infrastructure/software case.
```

Versión española:

```txt
El portfolio se organizará alrededor de cinco categorías principales:

1. Producción técnica AV / Eventos
2. Sound Engineering
3. Sound System Engineering
4. Redes AV
5. Infraestructura técnica

El objetivo es evitar que el portfolio se perciba únicamente como un portfolio DevOps/software.

Los case studies deben mostrar el posicionamiento híbrido completo:

- experiencia real en AV y eventos;
- sonido directo;
- producción técnica;
- redes y audio/vídeo sobre IP;
- infraestructura, automatización y sistemas backend.

Para v2.0.0 se priorizará una selección equilibrada de case studies que incluya al menos un caso AV/evento y un caso de infraestructura/software.
```
